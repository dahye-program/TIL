# 13장 전자서명

## 핵심정리

- 전자서명은 메시지의 생성자가 서명으로 동작하는 코드를 붙일 수 있게 해주는 인증 기법
- 전형적으로 서명은 메시지의 해쉬와 메시지를 생성자의 비밀키로 암호화한 것으로 구성됨

→ 메시지의 근원지와 무결성 보장

## 메시지 인증

### 메시지를 교환하는 두 상대방을 제 3자로부터 보호하는 것

그러나 메시지 인증만으로는 두 상대방 상호간을 보호하지 못함

= 둘 사이의 여러 형태의 분쟁 가능

1. Alice는 다른 메시지를 만들고, 그것이 Bob에게서 온 것이라고 주장
- Alice는 Bob과 자신이 공유하고 있는 키를 이용해서 쉽게 메시지를 만들고 인증 코드를 붙일 수 있음
2. Bob은 메시지를 보낸 것을 부인할 수 있음
- Alice가 메시지를 위조할 수 있기 때문에 Bob이 메시지를 보냈다는 사실을 증명할 수 없음

### 이렇게 송신자와 수신자 간의 완벽한 신뢰가 없는 상황에서, 인증 이상의 어떤 것 필요 ⇒ 전자서명!

## 전자서명 요구사항

앞의 속성과 공격을 근거로, 요구사항을 정리할 수 있음

1. **서명은 서명되는 메시지에 의존한 비트 패턴이어야만 함**
2. **서명은 위조와 부인을 방지하기 위해서 송신자에 대한 유일한 몇몇 정보를 사용해야 함**
3. **전자서명 생성이 비교적 용이해야 함**
4. **전자서명을 위조하는 것이 계산적으로 실행 불가능해야 함**
5. **기억 장소에 전자서명의 복사본을 유지하는 것이 실용적이어야 함**

## 직접 전자서명(Direct Digital Signature)

### 오직 통신하는 상대방(출발지, 목적지)들 만이 참여하는 전자서명 기법

(수신자가 송신자의 공개키를 안다고 가정)

**전체 메시지와 서명을 서로 간에 공유되는 비밀키로 암호화 → 비밀성 제공**

분쟁 시, 제 3자가 메시지와 서명을 살펴보아야만 함

만약 서명이 암호화된 메시지에 대하여 계산된 것이라면, 제 3자는 원래의 메시지를 읽기 위해서 복호키에 대한 접근이 필요

그러나 서명이 평문메시지에 대한 것이라면, 수신자는 평문 메시지와 이것의 서명을 차후의 분쟁 해결을 위해 보관 가능

## 전자서명의 표준

### DSS 접근 방식

DSS는 오직 전자서명 기술을 제공하기 위해 설계된 알고리즘을 사용

RSA와 달리, 암호화나 키 교환에 사용되지 않음(but, 공개키 기술임)

### RSA

- 서명되는 메시지 = 고정된 길이의 비밀 해쉬 코드를 생성하는 해쉬 함수 입력
- 서명을 생성하기 위해 해쉬 코드를 송신자의 개인키로 암호화
- 메시지와 서명 모두 전송

- 수신자는 메시지로 해쉬 코드 생성
- 송신자의 공개키로 서명 복호화
- 비교해서 일치하면 서명은 유효한 것으로 인정(송신자만이 개인키를 알기 때문에)

### DSS

DSS 또한 해쉬 함수 사용

- 메시지의 해쉬 코드와, 난수 k ⇒ 서명 함수의 입력
- 또한, 송신자의 개인키와 전역 공개키도 서명 함수의 입력으로 같이 들어감
- 서명 결과는 s와 r로 표시된 두 개의 요소

- 수신자: 메시지의 해쉬코드와 받은 서명, 송신자의 공개키와 전체 공개키가 검증 함수의 입력
- 검증 함수의 출력이 서명의 구성 요소인 r과 같은 값이라면 유효성 검증
